# 节点工作流

> 长久记忆：全局工作流的节点语义、运行规则、帮助来源与扩展边界。

## 界面

工作流页面由左侧模板列表、中部可缩放画布和右侧检查器组成。节点卡片始终显示一句用途摘要；问号按钮打开检查器的“用法”页。端口提示同时说明数据类型、必填规则与连接方式。

检查器提供“配置 / 用法 / 运行”三个标签。完整用法包含适用场景、输入与输出端口、配置步骤、典型连线、运行结果、常见错误和费用或副作用提醒。``WorkflowNodeDescriptor`` 与 ``NodeUsageGuide`` 是这些文案以及端口校验的统一来源。

LLM 与图片知识入库节点的服务商和模型可以留空，此时运行、校验与检查器统一继承设置页的“默认大模型”；节点显式选择的服务商和模型始终优先。

## 节点与端口

内置节点包括运行时输入、提示词模板、知识检索、创作知识准备、添加知识库、LLM、生图、生视频、条件分支、显式循环和结果输出。创作知识准备接收严格要素 JSON，逐项输出核验后的 `context` 和 `referenceManifest`；没有准确资料时暂停运行而不是失败。端口类型为文本、知识集合、图片、视频、音频、文件夹或任意值；普通输入端口最多接受一条连接，显式声明 `maxConnections` 的参考素材端口可接受多条有序连接。

运行时输入支持文本、提示词、知识集合、图片、视频、音频和文件夹。提示词输入节点保存分类与默认条目，知识集合输入节点保存默认入库位置；运行前可以为本次运行改选，并通过各自端口传入下游。媒体或文件夹通过 macOS 原生选择器选取，执行前校验类型和可读性，然后复制到本次运行的 `Assets/`，`run.json` 只保存相对路径。

提示词模板根据 `{{variable}}` 自动生成文本输入端口。模板可绑定提示词库条目；条目删除后继续使用节点保存的快照，并在检查器和运行记录中提示。

生图节点可切换“文生图”和“图片编辑”。图片编辑模式动态增加必填的原图端口和可选遮罩端口；切回文生图时，已经连接到编辑端口的连线会自动移除。遮罩必须遵守当前适配器展示的格式、尺寸和透明通道限制。

## 内置“添加知识库”工作流

``WorkflowStore`` 会为新安装和已有安装补齐稳定 ID 的“添加知识库”模板，并显示“内置”标识。内置模板可以查看、检查和运行，但不能重命名、删除或修改节点与连线；复制出的副本会取消内置属性并允许自由编辑。模板由“图片文件夹”“整理要求”“知识整理提示词”“知识集合”四个运行时输入节点、“LLM 图片知识入库”和“结果输出”组成。提示词和集合均通过显式连线进入入库节点；设置页“知识库添加”分类预置“产品图片知识整理”“汽车图片知识整理”和“人物图片知识整理”，工作流默认使用产品模板，运行汽车或人物资料时可临时切换对应模板；集合默认未分类。本次选择只写入 ``WorkflowRun``，不会修改节点默认值。旧版入库节点加载后会自动补齐这两个输入节点与连线，并迁移原提示词快照和目标集合。

入库节点递归扫描支持的图片，按稳定路径顺序逐张调用视觉模型。图片会缩放并转为 JPEG 后作为请求附件，原始文件不改变；模型返回的标题、正文和标签经过容错解析后，与原图一起写入 `KnowledgeBase/Files/<documentID>/` 和 SQLite。单次图片上限为 1–500，默认 50；单张失败不终止其余图片，最终摘要列出新增、重复、索引和失败数量。

若知识库嵌入服务商、模型和 API Key 均已配置，新资料会在同一次运行中自动建立索引；否则资料仍会成功保存并保持“等待索引”。相同原图与生成正文再次运行时跳过重复资料；同一图片使用不同整理要求产生不同正文时允许保存为独立资料。

``LLMChatMessage`` 的图片附件由 OpenAI Chat Completions、OpenAI Responses 和 Anthropic Messages 三种请求格式分别编码。服务商的 `supportsVision` 必须开启；API Key 仍只从钥匙串读取。

## 内置“汽车内广告分镜”工作流

稳定 ID `ADD0A11B-0000-4B00-8000-000000000002` 的“汽车内广告分镜”同样只读且可以复制。节点链为文章输入、创作规则检索提示、知识检索、要素提取提示、LLM 要素提取、创作知识准备、全片规划提示、中等推理规划、审校提示、高推理审校重写和结果输出，正常运行固定估算三次 LLM 调用，不自动生图或生视频。已有安装会自动补入规则节点与连线，并保留原有节点 ID。

规则检索层先根据文章语义从全局知识库召回可能适用的创作规则，再把同一份候选规则交给要素提取、分镜规划和最终审校。实现不解析某种 Markdown 表格，也不在 Swift 中写死人名或角色映射；规则可以覆盖人物、产品、车型、座舱、参考文件、禁止事项和连续性。只有正文明确声明为规则、强制约束、优先级或覆盖关系的资料才能改写文章身份；普通人物介绍、产品说明与相似案例不得越权。多条规则冲突时只执行正文明确的优先级；无法判定时保留文章明确身份并进入补资料流程。

要素提取按文章顺序逐个列出规则解析后的人物、目标产品/型号、指定车型或车内场景，同时在角色说明中保留文案称呼和映射。未指定车型时要求一条通用车内/座舱资料；未指定目标产品时产生阻塞缺口。知识准备对每项召回最多 12 个候选，并以完整资料的标题、标签和正文核验身份；相似但不同的人物、产品或车型不能替代。准确候选优先选择真实图片文件；Markdown、PDF 等文字资料不会被误当成卡片参考图。只有文字的准确资料仍可参与规划，但会记录没有卡片参考图的警告。所有知识检索在执行层对超长查询保留首尾并限长，避免导入长文章后超出嵌入模型上下文。

缺口把父 ``WorkflowRun`` 置为 `waitingForKnowledge` 并保存 ``WorkflowKnowledgeGap``。项目页按人物、产品、车内场景顺序打开真实“添加知识库”子运行，自动带入身份要求和对应的人物、产品或汽车图片整理提示词；用户仍需选择图片文件夹与可选集合并确认调用。子运行成功后重新核验全部缺口；父运行复用已完成节点输出，只从知识准备及其下游恢复。子运行失败、取消、索引未完成或补库后仍未命中时，父项目保持待补资料并允许重试；应用重启会把中断子运行标记失败而不破坏父等待状态。

命中的原图复制到父运行 `Assets/KnowledgeReferences/`，原知识库文件不改变。规划和审校模型只接收整理后的证据文字与参考 ID 清单，不接收原始像素。最终 Markdown 每镜用“参考资料”元数据声明引用；项目解析后会移除该内部行并把原图挂到卡片。

## 校验与执行

- 普通有向环在运行前被拒绝；每个允许的环必须恰好经过一个显式循环节点。
- 循环用 `seed` 开始、`feedback` 回传、`iteration` 驱动循环体、`completed` 输出最终值，最多执行 1–20 次。
- 条件节点支持等于、包含、不包含、正则、为空和非空；未命中出口记录为跳过。
- 整图运行执行所有可达输出；单节点运行先执行其上游依赖。多个分支按节点位置与 ID 的稳定顺序串行执行。
- 主动停止和失败会保留已经生成的部分结果；停止视频轮询不保证远端生成任务同步取消。
- `waitingForKnowledge` 不属于失败；等待运行没有结束时间，可在补库后恢复或由用户取消。

## 媒体适配器

``ImageGenerationAdapter`` 与 ``VideoGenerationAdapter`` 把不同服务协议统一为生成请求、进度事件和本地媒体产物。``MediaAdapterRegistry`` 提供源码内注册入口，并把支持的模型、尺寸、时长、参考素材和限制动态补充到节点帮助中。

内置 OpenAI Images 支持 `/images/generations` 文生图和 `/images/edits` 原图/遮罩编辑，OpenAI Videos 支持异步视频生成。PixMax 适配器原生支持最多 9 张图片、3 段视频和 3 段音频，按 `targetOrder` 稳定编号，并在提交前校验模型的画幅、分辨率、时长与音频能力。

API Key 和 PixMax Cookie 只从钥匙串读取，永不进入节点配置、帮助、日志或 `run.json`。未知 `adapterID` 会原样保留，以便安装包含该适配器的新版本后恢复，但当前不可执行。

## 持久化

定义保存在 `Workflows/workflows.json`，运行历史位于 `Workflows/Runs/<workflowID>/<runID>/`。`run.json` 保存节点状态和文本，`Assets/` 保存图片、视频、音频与本次选择的文件夹副本。历史不自动清理，也不进入普通设置备份。
